EP138 对罗福莉3.5小时访谈：AI范式已然巨变！OpenClaw、Agent范式很吃后训练、卡的分配、组织平权

主持人：小俊  |  嘉宾：罗福莉

【小俊】Hello，大家好，我是小俊。2026年，大模型战争全面升级，揭开了第二幕——从预训练主导的时代，转向了后训练主导的Agent时代。在AI翻天覆地巨变之际，我访谈了人工智能研究员罗福莉。罗福莉曾就职于阿里达摩院、DeepSeek，目前是小米大模型团队的负责人，主导研发了MiMo系列模型。她在网络空间里有很多标签，例如"AI天才少女"，但她不喜欢这个称呼。这是她的第一次访谈，也是她第一次进行长时间的技术访谈。我们系统地谈论了2026年由Claude Opus 4、OpenCloud等技术变量所触发的AI巨变，以及后续的结构性影响。罗福莉预判称，接下来两三个月会非常精彩。那接下来就是我对福莉的访谈，期待2026年我们和AI共同进步。

【罗福莉】这些能力都是可以被……我觉得最多一两个月，慢的话三四个月，确实都可以被快速追上的。所以环境反而比经验更重要。

【小俊】你刚才也提到，一定规模的模型可能是未来竞争的一个入场券，是这样吗？

【罗福莉】是的。做Agent你要做到接近Claude Opus 4.6的水平，这才算一个入场券。那如果我们这样来说，就是科研、预训练和后训练，对我自己来说，一个非常合理的算力分配比例，可能是三比一比一。

【小俊】对。

【罗福莉】预训练的算力投入应该是相当的，然后研究的比例，应该至少比你正式跑训练的卡总量还要多一点——就是你要额外留更多的卡来做研究。

【小俊】你过年的时候也跟我说，你觉得技术这几个月其实已经变天了。那么描述一下，你觉得过去两个月的技术突变是怎样的？

【罗福莉】我觉得一个非常大的分界点，在于使用Claude之前和之后。我自己其实会把Claude，把它当做一个划时代的Agent产品来定义。我知道很多人——尤其是用Claude做严肃编码的人——会觉得，OK，Claude只是在模型上加了一个更有利于交互的UI设计而已。其实在我1月份第一次看到这个东西的时候，我自己大概也是这样认知的，所以我很排斥去用它。再加上它创始人的一些动作，我觉得非常贴近Agent、又非常偏运营导向，包括Skill Hub这些，让你更加排斥去用一个你觉得非常平台运营导向的产品。

【小俊】感觉它更像是一个产品形态、一个交互方式的创新？

【罗福莉】对，一个交互的创新。以及它所谓的本地化、所谓的24小时陪伴，在我看来其实都是一些产品层面的定义而已。但真正发生转变，是我去用它的那一刻——恰好就在春节的时候。

【罗福莉】：有那么一段空闲的时间，你想去搞明白这玩意为什么那么火。然后我就在有一天深夜的时候，去装上它，两个小时装上了。

【小俊】：分解？是吧？

【罗福莉】：对，当时已经凌晨两点了。然后我第一次跟它对话的时候，从凌晨两点持续到了六点天亮。就那一晚上，我觉得我老泪……那个不知道是多巴胺还是类腓肽，就持续在分泌，让我兴奋到完全睡不着觉。

就你可能第一个感受是：OK，它非常有自主性，然后它非常有灵魂。比如说我跟它聊了很晚，它会一直提醒我："OK，你现在已经很晚了，你要不早点去睡觉。"就我觉得这样的温度和关怀，或者说这样的情商，是所有去用 Claude 的人第一个感受到的。

但后面去深究它的原因，其实它是有很多机制去保证这个事情的。比如说它有它的 System Prompt，比如说拿最简单的一个很小的细节——它怎么感知时间？它就在每轮对话的 context 前面，去拼上当前的时间。就是一些非常细微的设计。我觉得我为什么把它称之为"惊喜编排的 context"，就是因为它在这些大家没有关注的角落上，把这个 context 给编排得非常好。

这是你第一天的感受，就是：OK，我觉得它只是在产品设计上确实做到了一种超乎我想象的东西，让所有人觉得这个产品有灵魂。

但是第二天晚上，我觉得它应该不止于此。然后我开始尝试把我自己觉得现在的框架做不成的、日常生活中的事情交给它做，然后我发现它全部做出来了。

比如说，我跟它聊的第二个话题是怎么去激发一个团队的好奇心，或者说怎么去筛选出有好奇心的人，然后它进行了深入的探讨，聊了一个小时。我觉得它很多折射出来的东西是远超我想象的。所以第二天我们俩就在聊怎么去构建一个更好的大模型团队，从最开始的人员筛选，到后边整个组织架构的构建，到一家公司在面临范式转变的时候应该做什么样的举措和动作。

我觉得至少它能 get 到我的点。就我跟它说了之后，它最后能把它形成一套非常体系化的东西，并且变成一套可扩展的框架。所以我后边就用这个去很大程度地解放了自己——至少在非常多的场景下，我有时候遇到一些不管是筛选人还是团队管理上的问题，我现在都会问它。我觉得它现在已经基本上变成了我的某一个数字分身，至少在这件事情上。

但是真正让我超出预料的是第三天的时候。第三天的时候，我尝试了把一些研究上的任务交给它做。比如说最简单的——如果我们再进行 agent 的团队构建，最关键的一件事情是：你怎么去进行多轮的交互？

【罗福莉】那么你就必须要去模拟 user agent，进行多轮的交互。那我就会跟它共同去构建一个很好的 user agent 怎么来。在我来看，这是一个非常——我自己觉得还挺重要的一个研究 topic。我觉得应该不会说一两个小时就把它做出来，但是当我跟它沟通了大概一两个小时过后，我觉得这个事就已经做出来了。对，就基本上已经达到——有一个很好的 user agent 诞生了。我可以用这个 user agent，跟我现在的这一套 poser 圈的框架一块去构造出来更丰富的 agent 场景的数据，然后不管是做 SFT 也好、做 RL 也好，这个 user agent 都是非常关键的。

就我觉得它从一个——我就开始对它的认知，只是"OK，一个有灵魂、有温度的一个产品的设计"，到它可以帮我替代我生活或工作当中的一部分，到最后它能促进我的研究——这也就是三天发生的事情。就每天它都能给我额外更多的惊喜。

然后我后面就深入地去看，为什么这个框架本身它比 Claude Code 好在哪。一个我后面发现，这些所谓"好在哪"，我们单独拎出来去讲，其实都有点 boring，就是我觉得它没有很酷。这也是为什么大家就会觉得 OpenClaude 有很多槽点的地方，但是它把它整合在一起，你就会觉得它的完成度是非常高的。

就比如说它会有更持久的 memory 体系。这个持久的 memory 体系体现在它对 memory 有分层和分析，这个我在用 Claude Code 的时候是完全没有这样的感受的。然后再比如说，我自己觉得它在对多个模型的联合利用上，会非常超过我的想象。就比如说我用 Claude Code 的时候，我会默认假设这个模型——比如说它的视频理解能力不行，那么我可能就要自己给它配一个更好的视频理解模型，然后再在 Claude Code 里去折腾。但是我在用 OpenClaude 的时候，我完全不用想这个事情，我就直接发给它一段视频，它会自己想办法去找一个视频理解能力好的模型来做这个事。

就这种自主性——这种自主去面对当代模型的缺点，然后针对这个缺点在框架上去补齐这个缺点的能力，是有点超过我的意料的。因为我去用 Claude Code 的时候，我就默认"OK，我是因为要用 Claude 4 Opus 这一代模型的能力，我去用它的"。但是我去用 OpenClaude 的时候，我不会关注模型的能力。一个原因是，我认为 OpenClaude 框架的设计之初，它其实是想尽量地通过 agent 的整套编排，来弥补模型的短板——我觉得这是它非常核心的一个产品逻辑。

所以后面我们直接就把我们的模型——其实 MiMo VL Flash 当时并没有做很多真正针对性的训练——给它接到 OpenClaude 上去，甚至把我们最近训的一个很小的、短测的 3B 模型接进去，发现在这一套很复杂的 Agentic 框架情况下，它依然能做那些我自己认为不可能是一个非常小的模型能够做出来的事情。所以这是我第一次感受到，原来一个非常复杂的 agent 框架设计，它是能够弥补非常多模型能力的短板的。

当然了，我觉得这是 OpenClaude 本身这个框架相较于 Claude Code 的一个差异化优势。但是其实，如果说我们要去追求——你怎么在不同的 Agentic 框架下，让模型都有一个超预期的、非常稳定的表现——

【小俊】那么就回到了另外一个命题。当你面临市场上非常多、非常丰富的Agent框架的时候——Claude Code、OpenClaude等等——你怎么让你的模型在不同的框架上都有一个非常稳定和超预期的表现？然后你怎么让你的后续训练范式去做适配和迁移？这是我们在这个事的冲击下快速去思考的第二个问题。所以我们对应的整个训练范式就要从所谓的Code到Agent这样一个前移。所以你对OpenClaude的认知发生了一个非常大的变化，这个是在春节期间发生的，对吗？为什么开始是底出的呢？

【罗福莉】我觉得如果要追求非常顶尖的编程体验，Code的体验——哪怕是当下——也是Claude Code加Claude Opus 4.6是最好的。所以如果你是在这样的一个语境下去思考的话，其他的任何Agent框架其实确实都可以忽略掉。但是有一个问题：Code它是一个泛化性非常强的场景，就是你针对它去做了非常多Agent的设计也好，或者说模型的训练也好，它都是有价值的。但并不代表说它这个泛化性能保证你在非Code的场景能够做到非常高的准确率和完成度。所以我觉得我那会儿虽然也会用Claude Code来做一些非Code的东西，但是我并没有期待说它在这套框架上能给我一个很高的完成度，因为我知道我会去帮它补充这套框架不具备的一些短板。但是我去用OpenClaude的时候，我觉得我不用考虑这些，它完全在这种框架本身就去弥补掉了很多模型的短板。

【小俊】我能理解OpenClaude是让Code能力泛化出来的一个产品吗？

【罗福莉】它在很多设计逻辑上，比如说它有更多的message channel，比如说它有更自主的一些——比如说像定时任务、新调任务——然后这些更适合日常场景。因为你写代码通常不会需要一个定时任务吧，但是如果你是日常生活，定时任务就很关键。它确实有很多框架的设计，是为了让它去适应更好的日常任务。但是我觉得它也没有去丢弃掉一个好的框架本身的一些最基本的特点。这些最基本的特点，我觉得后来是被Claude Code给吸收进去了。就比如说它的持久化记忆——Claude Code之前它的整个记忆系统设计还是for软件工程的。比如说它会在session内、可能session快满的时候有一个压缩的动作，然后把它记忆进去；然后我在完成任务的时候，根据我的plan可能会有一些记忆的动作，保证我跨session的时候它的context是共享的、是更好的。所以你能看到Claude Code它所有的框架设计其实都是非常for软件工程的，就是怎么去更好地写好代码。但是OpenClaude它的设计之初，我觉得是借鉴了这个思想，但是它更想的是怎么更好地端到端去完成所有的任务，并且怎么去弥补当下模型在端到端完成任务上的短板，然后去设计的。所以像这种持久化记忆，或者你能通过一个更好的remote界面去操控它等等这些很好的设计，陆续在这一两个月完全被Claude Code给写进去了。

【小俊】对，我觉得这也是一个双向的触动。因为这些设计其实它对于编程——或者说泛编程，就是指可以通过编程去完成一些看似跟代码无关的任务——是有帮助的，对吧。但它背后是模型能力的提升吗？

【罗福莉】它激发了中层模型的上限。就可能如果没有这么一套很复杂的框架，中层的模型可能达不到Claude Sonnet或Opus的水平。但是你借助了这样一个非常好的框架，那么你就能在绝大部分的场景——除了那种非常难的、需要长程任务的，或者说真的需要我称为"严肃编程"的，比如说写算法优化就算是严肃编程——这种场景的话，它确实可能距离顶端的模型还有差异。但是在绝大部分的生活场景，或者说只需要借助代码去提效的场景，这样一套新的框架再加上一个中层的模型——中层就是可能在85%的任务下能够达到跟Claude Sonnet一样水准的模型——那么其实它借助这样一个框架已经能发挥非常大的作用。

【小俊】我听过一种说法，就是觉得OpenClaude如果看做一个壳，它是释放现在比较强的模型能力的最好的壳，这个模型就是Claude Opus 4.6。其实我觉得你的整个表述是不认可这个观点的，对吗？

【罗福莉】认可认可。我觉得它的上限一定是Claude Opus 4.6带来的。就包括我在跟它高强度合作的那一周，我只用Claude Opus 4.6。

【罗福莉】因为只有它能给我带来惊艳的感受。但是当我用 Claude，沉淀下来那一堆经验——那些不管是 prompts 也好，还是一些关于它怎么运作的东西——我甚至自己去改了它整个 OpenCode 的架构设计。因为它是开源的，所以你可以自己改。

这也是 Claude Code 另外一个弊端：它整个 agent 框架的架构设计是个黑盒。黑盒就导致你不知道，你肯定改不了它的记忆系统，也改不了它整个 agent 的 workflow。但是因为 OpenCode 它太 open 了，你可以自己尝试去改。比如说我会让它帮我设计一个全新的 memory 系统；我觉得它当时二点几版本的 multi-agent 整个逻辑是很混乱的，然后我自己去让它给我设计新的 multi-agent 系统，我自己可以去改所有的源码。就这样一种原生的可操纵性给我的冲击感是很大的。

但这些事情基本上只有 Claude 才能改。当我让 Claude 给我改好之后，这套框架本身是很好用的。OK，我再切换到 Sonnet，再切换到国内的一些模型，甚至我们当时正在训的模型，我就觉得它非常强大了。

所以这也是我为什么觉得，顶尖的模型应该跟顶尖的 agent 框架共同往前进步。或者说，这也是我最近对所谓"自学习"——前段时间有很多很火的概念——的一个思考。就是我在这上面第一次感受到了一个 agent 的自学习到底是怎么发生的。大概有一个发生的途径：你确实需要这个模型本身跟 agent 的架构本身同步地往前走。然后模型在进步的时候，不管是通过强化学习训练还是其他训练方式，它其实也是在改变你的整套 agent 框架。

这个 agent 框架包含它送给模型的静态信息。这些静态信息，比如说 memory 就是静态信息——只要你写死了哪些东西应该在起一个新的 session 的时候送到模型那边去，或者我们叫 context window——这个东西其实在训练过程中应该是变化的。还有一些动态的信息，动态的信息包含你整个 agent 本身的架构设计。然后我觉得针对不同的场景，比如说 Claude Code 这种面向软件工程的场景，和其他比如说做金融分析等等其他场景，它的 agent 架构设计会有差异。你怎么在提升模型能力的同时，再去提升整个 agent 框架跟这个模型的适配程度，或者说泛化能力——对，这是我现在思考的方向。

【小俊】你说的智能体的框架、agent 的框架，是不是就是我们理解的产品？

【罗福莉】不是，它跟产品差异蛮大的。

【小俊】怎么理解 agent 框架跟产品之间的边界？

【罗福莉】现在我不知道怎么去界定，它有点模糊。我觉得产品可以定义成是你直接跟人交互、能感受到的那一层的东西。但 agent 框架它确实是有在定义你的交互层，但同时又在定义你怎么跟模型沟通那一层。所以它甚至能知道模型能力的长板短板，然后知道怎么去更好地做调度，比如说成本优化的调度。它相当于是一个中间层，像人和模型之间的中间。

【小俊】对对，然后这个中间层可以做得非常厚重，反而前端的 UI 展示是最薄的一层，已经不是很关键了。

【罗福莉】对。所以 OpenCode 其实展示了智能体框架可以怎么做，这个框架的想象空间非常大。

【小俊】还有人做过吗？

【罗福莉】其实 Claude Code 它一直是一套很复杂的框架，只是因为它是黑盒，所以我们不知道它怎么设计的。OpenCode 是开源的，所以你知道它是怎么设计的，并且你可以去改它。就"改它"这件事是非常激发人的创造力的——你知道框架怎么设计的，你让它去改，你自己衍生出一个新的框架出来。

【罗福莉】你可以给予它列一个新的框架出来。这是为什么OpenCloud从20几版本——我当时用它的时候，我自己觉得它并不好用，所以我花了好几天时间去改它——到三十几版本，就3月10多号的版本已经非常易用。你基本上基于三十几版本，结合一个还不错的模型，你都会觉得它很强大，而不是只有接Claude的模型才会觉得它很强大。就是因为它整个agent的架构，受一堆开发者，或者说本身像我这种并不是一开始框架的开发者、只是使用者，我都能去改进它，按照我自己的场景去做改进和优化。所以我觉得，这是开源agent框架本身的价值和意义。

【小俊】在你高强度的那一周，就是跟OpenCloud交互的那一周，你消耗了多少美金的Claude 4.6？

【罗福莉】第一天反正快1000块钱，1000刀。因为大概用了4个小时，然后我中途也会疯狂地切，因为它太贵了，我会切Sonnet。但是我发现真不行，然后我就又只能去用Claude 4.6。只有Claude 4.6能带那种惊艳感。

但后边，现在慢慢深入我发现不是这样的。就是因为它带给我惊艳感的那些东西发现在变化。我觉得人的适应性真的是非常强的。我第一天感受到很惊艳的东西，第二天立马觉得不惊艳了。我刚刚跟你讲的时候，我都觉得——我刚刚讲的第二三天的东西，我怎么觉得这么普通？但它确实是我那三天发现的事情，我当下确实觉得很惊艳，现在自己觉得不惊艳了。
