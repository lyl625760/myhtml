【Kimi解读】：我自己的研究，主要就是想把 Language Agent 这个东西真实地做出来。特别是能让 Language Agent 学会操控电脑，或者操控浏览器，能帮助大家完成一些日常的工作，起到一个助理的作用——我们现在管它叫 Computer Use Agent。

形象来说，我的研究在于如何让这个 Agent 学会去感知环境、进而理解环境，在此基础上进一步生成动作，也就是 action，来和环境进行交互。

一开始我们的难点在于，如何让这个 Agent 去理解互联网或者网络这样一个环境。那时候大家只有单一模态的 Language Model，Language Agent 只能去看 HTML 并且理解它，但这个内容非常冗杂。所以我就研究如何让 Language Agent 能够去理解 GUI 界面，进而更高效地理解环境并且生成 action。

后来我发现，Language Agent 虽然能够理解环境，也知道在 GUI 上有什么样的功能，并且能够准确地用自然语言描述下一步是什么样的 action，但往往它并不能把这个 action 很好地转化成一个可执行的操作。所以我进一步做了一个叫 Grounding 的工作，能够让 Language Agent 生成的 action 准确地转化成一个 Program，并且在浏览器上进行执行。

在这个基础上，我们逐渐能够帮助 Language Agent 在单一的步骤上做到很好的效果。但我发现，一旦让它做一些非常长、非常复杂的任务，效果就会变得差很多。随后我开始研究如何让 Language Agent 能够做比较长的规划和推理，并且提出了如何用 Language Model 内部的推理模型去辅助 Language Agent 做这种规划。

我最近在 AI 公司实习，在研究如何用大规模的强化学习，把 Language Agent 的效果做得更强。

【小俊】：我们今年讨论Agent非常的多，能不能请博元先给大家讲一讲，怎么给Agent下一个定义，并且给Agent做一个常见的分类？

【Kimi解读】：首先我觉得"Agent"这个词现在用得比较笼统，有各种各样的用途。但就我个人的认知，我会首先把它叫做"Language Agent"，因为目前大家的Agent都是基于一个Language Model去构建的，并且这些Agent之间会用语言作为沟通交流、推理和规划的载体。

在这个基础上，关于这个定义有一个常见的问题：Language Agent和Language Model有什么核心的区别？因为它们看起来是非常相似的东西。我觉得核心其实就在于——这里是否存在一个"环境"。Language Agent需要学会很好地和环境做交互，具体体现在两个点：第一个点是它能够很好地感知环境，我们管它叫Perception；另一个点是它能够很好地生成并执行Action，从而影响环境。

我们可以具体对比一下Language Model和Language Agent在训练以及推理时的区别。对于Language Model，它接受的是一个Input，然后输出一个Output，这个过程就结束了。但对于Language Agent，它的形式是这样的：首先，Language Agent会感知环境，从环境中获得一个Observation（观察）。这个Observation的类型会随着Agent所属的环境以及应用场景有所不同。比如说，如果是一个Coding Agent，它的Observation就会是整个Codebase代码库，以及当前执行的记录或环境的反馈；如果是一个Computer Use Agent，它的Observation就会是当前浏览器或电脑显示屏的GUI截图，或者对应的HTML。

【小俊】：我们今天的重点就是想给大家读几篇关于Agent的最近论文。好久都没有给大家读论文了。就在这两周，OpenAI、Kimi、阿里，包括前两天的智谱，都发了新的模型，他们都是在为Agent工作流而设计的模型。另外，Manus的首席科学家也发布了文章，讲述构建Manus的经验与心得。今天我们就想请博元来讲解这几个和Agent相关的技术报告和博文。延续我们之前的传统，我先来问博元几个问题，然后再把时间交给他。

【Kimi解读】：好的，关于我自己的背景，我即将开始博士第三年。我的PhD研究和我之前的经历其实有一个很有趣的衔接——我本科和硕士阶段研究的领域叫做自然语言处理中的"信息抽取"。这个领域想做的事情是：把非结构化的语言，比如我们平时日常的聊天或者一篇文章，通过一些NLP的技术，把它变成一个结构化的表述，比如整理成一张图或者一张表，然后在这个基础上进一步构建像ChatGPT这样的Agent。

所以当ChatGPT出来的时候，我们整个研究发生了很大的变化。但我也在这个基础上发现，ChatGPT，或者说Language Model整体上，已经能够很好地作为人类语言和其意图之间的一个界面。所以我就开始想，能不能在这个基础上，利用这样一个很好的语言界面，去进一步做一些更难的、或者对人类更有帮助的事情。

【Kimi解读】然后这个Observation会被作为模型的Input的一部分，会被送给Language Model去做处理。当然这个过程中还会有Agent需要去完成的Task，还有之前环境交互的内容。就比如说假如我们这个任务是让这个Agent去帮我们订一张机票，那么这个Task就是比如说订一张从北京到上海的机票，然后它之前的交互内容就是这个Agent之前已经执行过的操作，就比如说在输入框里输入什么样的内容。在这个基础上，这个Language Model会基于这样的一个Input随之对它进行执行，然后环境的状态也可能会发生变化，就比如说我们这个页面刷新了，或者是对于Coding Agent来说，我们获得了一个新的执行结果，它可能是有Bug，那我们就需要进一步去Debug、改变这个执行的结果。这个基本上就是我认知中对于Agent的一个定义。

当然我觉得Agent它现在跟应用非常相关，我们也可以基于它应用的场景对它做一个简单的划分。就目前大家研究比较多的Agent主要分这几类：首先我觉得目前做得最成熟的应该就是Coding Agent，就比如说Cursor，还有前段时间那个很大的收购案里的Windsurf，它们里面都包含Coding Agent。具体来说这个Coding Agent是什么呢？就比如说我在写代码的时候给一个指令，比如说"请帮我写一个什么样的API接口"，然后这个Agent它会写一段代码，然后自己执行，执行之后看一下这个执行结果，如果有Bug的话，就自动去给它Debug，最后生成一个很不错的API接口。这个也可以拿来做Coding、做Debug一些有问题的代码。

整体上我感觉这些Coding Agent现在效果已经非常强大了。就包括我自己写代码的时候，我觉得就像是不停地有很多实习生在帮我写代码，效果体验非常强大。甚至有时候我能同时开好几个Agent去帮我，比如说在三五个接口上同时做好几个任务，这个对生产力的帮助非常强大。

然后还有另外一种比较强大的Agent叫Search Agent，典型的例子就是OpenAI的Deep Research，包括Gemini以及各种公司，他们都有出对应的产品和模型。这种Agent所做的事情就是给它一个指令，它们接着会对这个指令进行大规模的检索或者搜索。比如说我们给它一个指令像是"请调研一下当前市面上所有的GPU提供商，它们的价格、它们的策略以及能够提供多少GPU"，那么这个Agent就会先对Task做理解，然后再使用Google Search或者一系列的搜索引擎，或者通过访问一些URL进行大规模的搜索检索，然后生成信息，最后返回一个Report。我觉得这种Agent对于市场调研之类的场景会有非常强大的帮助。

还有两类我觉得最近做得更多的Agent：一个是Web Use Agent，这也是我们这次会讲的Kimi K1.5，还有Claude 3.7，它们的实验中非常强调的一个场景；还有一个就是Computer Use Agent，像我们这次会讨论到的OpenAI的ChatGPT Agent、OpenAI Operator，它们是属于这个类型的。它具体做的事情就是给一个Task，比如订机票，或者说在网上预约一个餐厅之类的，然后这个Agent自己会生成一串的Action去完成这个任务。

像这一类的任务，其实在一些相对比较传统的行业，就比如说RPA——流程自动化，像Salesforce这些公司，它们需要做的事情就是给一个固定的流程，然后需要手动写一串的Workflow去完成这个事情。但是现在有了Agent，或者是基于Language Model的一些其他Agent，我们可能能够完全自动化地做这个过程，所以就不需要大量的工程师来手动写Workflow，反而是可以用一个Agent自动地把这个过程全部完成。所以这对RPA这个产业也可能会产生一些影响。

所以我觉得这几个Agent应用场景的不同会有很大的区别，但从它们的任务建模上来看，它的区别主要在于它的Observation Space，就是观察空间。

【Kimi解读】比如说，Coding 是 Code Agent，Search Agent 是一个 Text Browser，Computer Use Agent 是一个 GUI 的 Browser。它们的 Action Space 也不一样：像 Coding 的话，它的 Action 就是生成一串代码或者修改一串代码；而 Computer Use Agent 就是所有的浏览操作，比如说点击、或者在某个输入框里面输入什么内容。

【小俊】具体来说，这次的 Kimi K2、Kun、San、Coder、Cognify Agent，以及应用公司 Manus，它们之间的技术方案有差异吗？

【Kimi解读】有的。我觉得可能比较大的差异，还是 Manus 跟其他几个模型之间的不同。就像 Manus 在那个 Blog 里面提到的，这个区别就在于：我们是否利用这个模型的 In-context Learning 的能力，还是从头 End-to-End 地去重新训一个 Agent。

Manus 就属于非常详尽、非常精细地去利用好了语言模型 In-context Learning 的能力。它们里面不太会涉及到模型训练，但是可以通过很好地调节 Prompt，以及设计一些 Multi-Agent 的系统，能够非常快地迭代，做出一个能够让大家直接拿去用的产品，并且能快速地做迭代。

这种方式下，它非常容易迭代，能非常容易地调节 Agent 的各种能力，并且能很好地做 Multi-Agent 的场景。比如说，当我们设计这个系统的时候，我们会认为有不同的角色、不同的 Model，在里面承担不同的职能。就比如说我们要写一个网站，那么不同的语言模型或不同的 Agent 就有不同的职位：我可能会需要一个"产品经理"角色，很好地去设计这个网站的外观，或者它对用户来说有哪些比较重要的属性和 Feature；同时还会需要一个很会写代码的 Agent，比如说现在大家一般会用 Claude；在这个基础上，可能还会有一个专门来 Debug 的 Agent，或者是来评判代码写得怎么样的 Agent。所以这是一个 Multi-Agent 的系统。

这种 In-context Learning 的范式让这样的操作非常方便，同时现在不同的模型可能有不同的长处，所以基于这个方式能够很好地调出一些很好用的产品。

但是，如果我们有一个非常明确的场景，比如说 Coding 或者 Browsing、Computer Use，那如果能专门训练一个模型或 Agent 出来，它会变得更加强大。

【小俊】那端到端训练的模型呢？

【Kimi解读】端到端训练的模型，如果我们能以某种方式很好地获得大量的 Reward Signal，或者大量的 Training Data——我们一会儿讲 Kimi K2 的论文会讲到——那我们就可以在某个特定场景下训练出一个非常强大的 Agent。它的好处在于，在某个特定场景上会非常强大。

但是我觉得，目前这种端到端训练的方式，如果要做 Multi-Agent 的话可能会相对难一些，因为数据可能会相对比较难获得。如果我们要训这样一个 Agent，数据上面需要两方面：一方面是一些 Synthetic 的数据，就是这个 Agent 的 Action 和 Environment Feedback 相互交叠的数据；而一旦到了 Multi-Agent 的场景下，这种数据将会非常难生成或者非常难获得。

当然，我们可以用 RL 训练，也就是强化学习——给它一个任务，然后这个 Agent 去做探索，最后评判它是否成功。但这样的话，在 Multi-Agent 的场景下，它的执行轨迹会非常非常长，而且里面多个 Agent 每个都有自己的身份和角色，那我们就很难分清楚最后的 Reward 应该归属于哪个 Agent，以及应该怎么把训练信号传导回去。

【小俊】为什么我听到大家对于 OpenAI 的 Agent 整体是比较失望的？整个 AI 社区对它的评价是什么样的？

【Kimi解读】对，这次……因为我前段时间刚从 ICML 回来，ICML 上大家讨论得非常多。我觉得大家的反馈可能根据不同的……

【Kimi解读】大家不同身份会有不同的感受。就是做科研的朋友们，比如像我这样的PhD，还有一些更senior的研究人员，大家会有一种"松了一口气"的感觉。

【Kimi解读】对，松了一口气。因为我们一直会担心，哪天OpenAI或者Google，他们可能一次性通过scale up这种RL training，或者是scale up在环境里面的exploration，能够在一定程度上把这个问题变成一个"wild solved"的问题。因为这种事情其实已经发生过几次了——就像我前面提到的2022年ChatGPT的冲击，基本上把NLP的很多小方向给干掉了，比如说我自己以前的研究方向。我们发现大规模的训练有时候能够把一些细分的小任务很大程度地解决掉。包括后来OpenAI发了GPT-4V，这个vision模型把多模态这边的研究又很大程度上影响到了。然后我们看到ChatGPT agent，也就是OpenAI他们的agent，主要是Operator和Deep Research，像这种方式下面能看到可能会有比较明显的scale up方案，能够很大程度上解决这个问题。所以大家都有点提心吊胆。我当时就感觉，如果他们一下子发现特别强的agent，可能对研究方向会有比较大的转向。但我觉得这个问题确实会比较难以解决，所以里面还有非常大的空间可以去探索。

然后除了research之外，另外还有startup，就是创业公司，他们极其兴奋，甚至很多人在做一些有趣的小demo、小视频，把他们自己的agent拿出来跟OpenAI的ChatGPT agent做对比。比如说我看到一些很有意思的视频或demo，他们搞了一个类似"斗兽场"或者像足球赛一样的形式，在一个显示屏上开两个框，左边是他们自己的agent，右边是ChatGPT agent，然后两边同时开始做ChatGPT agent里面展示的同一个task，主要是flight booking。因为其实可以很容易地通过收集一些领域内的数据，或者做一些特定的prompting，让这个agent很擅长做这件事。然后就看他们不停地解说："ChatGPT agent现在还在执行这个操作，但是我们的agent已经完成了！"就是这种感觉。所以整个startup那边就特别开心，因为这可能证明这个问题比较难解决，里面还有很多探索空间可以做，对他们融资或者产品都会有很大帮助。

【小俊】相比来说，国内为什么这次反响比较大？以及像阿里、千问和Kimi发布的这个agent模型，他们的相似和不同是什么？

【Kimi解读】对，我觉得Kimi这次release的时间真的非常非常好。这应该是比较早的一个、比较focused关注在agent能力上的模型。当然之前有很多模型，很多language model里面已经包含了这样的agent能力，但这次Kimi在官方宣传上也主要强调这是一个"open agentic intelligence"，他们的paper title就是这样写的。然后正好最近大家对agent的能力比较感兴趣，agent本身也比较受欢迎，所以一下子就非常热门。

另外一个是，我觉得这个模型本身已经足够强大了。因为现在整个预训练数据方面，Kimi或者千问他们已经有很大的积累，内部有大量数据，再加上各种训练算法或者架构上的成熟，它本身能力非常强大。我整体使用的体验，就是一个小一号的GPT——一个完全开放的、小一号的GPT。这样相当于大家就可以免费获得一些非常强大的agent了。然后我觉得大家可能拿这些模型去做一些小应用，比如说写一些网页前端之类的，会发现它效果确实很不错。

【小俊】所以我覺得這次這可能是比較受歡迎的一個原因，因為Agent從大腦走出來，要跟世界發生交互了，甚至和物理空間發生交互。那你覺得Agent的安全問題，是值得更多考慮的嗎？

【Kimi解读】對，是的，我覺得這個是非常重要的一個問題。我其實正好昨天剛released了一篇論文，就是在研究這個問題。

首先我想先講一下，為什麼Agent Safety會有這樣的影響。Agent它執行的action，可能對世界會產生一些難以逆轉的影響，這個也取決於Agent的類型。比如說如果是Deep Research這個類型的Agent，它可能只是在網頁上做「讀」這個操作，所以它不會對這個世界、或者對網站本身產生影響，這是非常安全的。就算它生成了一些很離譜的action、做一些很奇怪的事情，也不會對別人產生影響，所以完全沒有問題。

但是如果我們看到Computer Use Agent，比如說Operator，那它可能是實實在在地在影響這個世界。我自己有一個親身的例子——大概一年半之前，當時我做了一個工作叫TheAct，應該是當時市面上相對最早一個把Web Agent真的在live website上跑起來的Web Agent。這個Agent跑完之後我在做demo，有一個demo的任務就是說：幫我在特斯拉的某家門店訂一個試駕。我做這個demo的時候其實沒有特別在意，但這個Agent它真的做成了。做成之後我一開始也沒在意，然後隨後聖誕節的時候，它就不停地給我發郵件，說「你是時候來提車了」。
